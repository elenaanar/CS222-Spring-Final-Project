## Bonus, Up To 5 Points

Bonus points are subjective and discretionary. They are awarded for work that is unusually impressive, creative, useful, technically strong, or thoughtfully executed beyond the required rubric. Bonus cannot compensate for missing required deliverables.

## Detailed Stage 1 Rubric: Initial Agent + Workflow Design

### 1. Proposal-Writing Understanding: 8 Points

- 8: Workflow shows a strong understanding that proposal writing includes problem framing, motivation, novelty/gap, prior work, method, figure planning, evaluation, risk, feasibility, sources, and revision.
- 6: Workflow covers most major proposal-writing components, but one area such as novelty, evaluation, or revision is underdeveloped.
- 4: Workflow focuses mainly on drafting sections, with limited attention to proposal logic or revision.
- 2: Workflow treats proposal writing as simple text generation.
- 0: No clear understanding of proposal-writing process.

### 2. External Research Quality: 6 Points

- 6: Video or demo artifact includes useful funded proposal, PhD proposal, NSF/fellowship, grant-writing, proposal-template, or agent-workflow resources; each source is connected to a concrete design decision.
- 4: Sources are relevant, but the connection between sources and workflow design is only partially explained.
- 2: Sources are listed but shallow, generic, or not clearly used.
- 0: No meaningful external sources.

### 3. Initial Workflow Structure: 6 Points

- 6: Workflow has clear stages, inputs, outputs, transition rules, project state schema, stopping criteria, and artifact outputs.
- 4: Workflow structure is mostly clear but missing one important element such as stopping criteria, state schema, or transition rules.
- 2: Workflow has named stages but weak input/output definitions.
- 1: Workflow is mostly a list of prompts or UI screens without process logic.
- 0: No coherent workflow structure.

### 4. Initial Agent Or Prototype: 5 Points

- 5: Vibe-coded initial agent or prototype demonstrates the workflow idea with a real interaction, even if incomplete.
- 4: Prototype demonstrates the main workflow but has limited interactivity or weak state handling.
- 2: Prototype is mostly static or only shows a prompt sequence.
- 0: No initial agent, prototype, or interaction artifact.

### 5. Stage 1 Presentation Video: 5 Points

- 5: 5-minute video clearly explains the workflow, rationale, external sources, initial demo, example user journey, and planned Stage 2 refinements.
- 4: Video explains the main workflow and demo but lacks one important element such as sources or refinement plan.
- 2: Video is difficult to follow or mostly shows UI without design reasoning.
- 0: Missing video or recording.

## Detailed Stage 2 Part 1 Rubric: Refined Source Code

### 1. Meaningful Refinement: 6 Points

- 6: The refined agent/workflow makes a meaningful improvement beyond Stage 1, such as better state schema, better source handling, better revision cards, better rubric parsing, or better evaluation.
- 4: The refined version improves the baseline but the improvement is narrow.
- 2: Changes are mostly cosmetic or only lightly adjust prompts.
- 0: No meaningful refinement beyond Stage 1.

### 2. Implementation Completeness: 6 Points

- 6: Source code implements the main refined workflow behavior and includes the key components needed to review the agent or workflow.
- 4: Source code implements the main idea but one important component is incomplete or hard to inspect.
- 2: Source code is present but mostly skeletal, static, or disconnected from the stated workflow.
- 0: No usable source code is submitted.

### 3. Code Organization And Clarity: 4 Points

- 4: Code is organized, readable, and easy to review.
- 3: Code is mostly understandable but has some confusing structure or naming.
- 1: Code is difficult to follow or scattered across unclear files.
- 0: Code cannot be meaningfully reviewed.

### 4. Proposal-Workflow Alignment: 4 Points

- 4: Source code clearly supports proposal-writing needs such as framing, novelty, method, evaluation, feasibility, source handling, or revision.
- 3: Source code supports proposal writing but misses one important requirement area.
- 2: Source code is only loosely connected to proposal-writing needs.
- 0: Source code does not reflect the proposal workflow.

## Detailed Stage 2 Part 2 Rubric: Final Proposal

The proposal is graded as a final PDF. The source file is checked for reproducibility, but the PDF is the official proposal artifact.

### 1. Format And Submission: 5 Points

- 5: Submitted as `proposal.pdf`; maximum 3 pages excluding references/appendices; readable 11-point font; 1-inch margins; complete required sections.
- 3: Mostly correct format but one minor issue such as missing keywords, unclear section names, or small formatting inconsistency.
- 1: Major format issue such as too long, hard to read, missing several required sections, or not clearly a proposal.
- 0: No usable PDF submitted.

### 2. Figure Or Visual Explanation: 7 Points

- 7: Includes a clear, relevant figure, diagram, architecture sketch, workflow chart, or evaluation design with caption. The visual helps the reader understand the project and is referenced in the text.
- 5: Includes a useful visual with caption, but the text does not fully explain it.
- 2: Includes a relevant visual, but it is too generic, hard to read, weakly captioned, or only partially connected to the proposal.
- 1: Includes a decorative or low-quality picture that does not explain the project.
- 0: No figure or visual explanation.

### 3. Motivation And Problem Framing: 6 Points

- 6: Problem is concrete, scoped, and tied to a clear user, setting, or research need.
- 4: Problem is understandable but broad, generic, or missing a specific stakeholder.
- 2: Problem is vague or mostly restates the project idea.
- 0: No meaningful motivation or problem statement.

### 4. Novelty And Relation To Prior Work: 10 Points

- 10: Clearly explains relevant existing work, tools, or approaches; identifies a specific gap; states what is new or different about this project.
- 8: Mentions prior work and a plausible difference, but the novelty is not fully precise.
- 5: Has a general novelty claim but weak comparison to prior work.
- 2: Uses vague claims such as "better" or "smarter" without evidence or comparison.
- 0: No novelty, gap, or prior-work discussion.

### 5. Method And Workflow Detail: 9 Points

- 9: Method is concrete and reproducible. Agent stages, inputs, outputs, project state, human interaction, LLM calls, and revision loop are described.
- 7: Method is mostly clear but missing one important operational detail.
- 4: Method is plausible but too high-level to implement or evaluate.
- 2: Method is mostly a promise to use AI without a workflow.
- 0: No meaningful method.

### 6. Evaluation Plan: 7 Points

- 7: Evaluation includes concrete test scenarios, metrics or rubric checks, before/after comparison, and unsupported-claim checks.
- 5: Evaluation has reasonable scenarios but weak metrics or unclear success criteria.
- 2: Evaluation is vague, subjective, or only says the result will be inspected.
- 0: No evaluation plan.

### 7. Feasibility, Milestones, Risks, And Resources: 4 Points

- 4: Research plan is credible and appropriately scoped; milestones, timeline estimates, resources, risks, and mitigations are specific.
- 3: Mostly feasible but milestones, timeline, or risks need more detail.
- 1: Scope seems too broad or resources are unclear.
- 0: No credible feasibility plan.

### 8. Writing Coherence And Polish: 2 Points

- 2: Proposal is coherent, concise, and internally consistent.
- 1: Understandable but contains awkward transitions, repeated claims, or inconsistent terminology.
- 0: Difficult to follow.

## Common Deductions

- Unsupported claims are stated as facts: deduct up to 3 points.
- Stage 2 Part 1 source code is missing or unrelated to the project: deduct up to 5 points.
- Figure is unreadable in the PDF: score at most 3 for the figure criterion.
- Missing references or assumptions section: deduct up to 2 points.
